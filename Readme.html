Product Overview

AML (Agent Modeling Language) Profile for StarUML is an implementation of the UML 2.0 Profile for AML in the StarUML™ CASE tool. This profile allows users of StarUML to create and edit AML models supporting the analysis and design of software applications that draw on multi-agent systems theory and engineering practice. For more information relating to AML see:

R. Cervenka and I. Trencansky. Agent Modeling Language. Language Specification.Version 0.9. Technical report, Whitestein Technologies, December 2004. (PDF)

or visit http://www.whitestein.com/pages/solutions/meth.html.

2. Implemented AML Features

The AML Profile for StarUML provides stereotypes, including their iconic notation and specific extended notation (if applicable), for most AML modeling elements. Furthermore, it defines sets of tagged values, new types of diagrams and element pallets placed in the StarUML Pallet Toolbar.

3. Deviations from the AML Specification

To resolve some problems with StarUML and particularly with its implementation of UML 2.0, and also to provide some late improvements for AML v1.0, the implementation of the AML profile does not completely adhere to the AML v0.9 language specification. The following list is a summary of all implemented deviations:

  • Serviced Property is not explicitly supported. Each property (Part or Port) connected to a Service Provision or a Service Usage is considered to be a Serviced Property implicitly.
  • The icon for the Decidable Goal has been changed to differentiate it from UML Activity Final or Final State.
  • Stereotypes «sub», «join», «create attribute», and «destroy attribute» are based on Messages and Stimuli as Dependencies cannot be used in StarUML interaction diagrams.
  • Join specifies the selector by means of the Branch property instead of a tagged value.
  • Interaction Protocol is modeled as a parametric Collaboration instead of parametric Interaction which is not supported by StarUML. Roles and template parameters of a Collaboration are used to specify formal template parameters of an Interaction Protocol. The specified template parameters can, but need not, also explicitly contain the Collaboration roles. The usage of an Interaction Protocol is therefore modeled by means of a Template Binding combined with a Collaboration Use.
  • Service Specification is modeled as a stereotyped Collaboration. The service parameters are extracted from the Service Protocols into their comprised Service Specification which commonly specify parameters for all comprised Service Protocols. Therefore, parameter names of all Service Protocols within one Service Specification should be harmonized, i.e., one parameter name should represent the same value in all Service Protocols. Both the provider parameters and the client parameters are specified as Template Parameters within a Service Specification. These two sets are delimited by the '-' special-purpose parameter. The provider parameters are specified first, then the '-' parameter, and finally the client parameters.
  • Service Specification shows the comprised Service Protocols only in the textual notation. See Fig. 5-73 in the AML specification for more details.
  • Hosting Association has the stereotype «host», instead of «hosting» as defined in the AML specification, because the keyword must be different from the the stereotype of Hosting Property.
  • The stereotypes «attribute hosting», «resource hosting», «environment hosting», and «organization unit hosting» were added to provide specific icons for the hosting of entities of these particular types.
  • The stereotype «x» was added to a Frame in order to model the AML diagram frame. This is a normal Frame extended by the possibility to show the formal parameters of Interaction Protocols and Service Protocols.
  • Mental Constraints of Mental Properties and Constrained Mental Elements, defined in AML as tags, are specified as constraints with the corresponding name, i.e., commit, pre, commpre, inv, cancel, and post.
  • New stereotypes applied to Attribute were added to allow extended notation for Parts of specific types (see Sect. 2.2. in the AML specification). The following stereotypes were added: «ag» for Agent Type, «re» for Resource Type, «en» for Environment Type, «ou» for Organization Unit Type, «er» for Entity Role Property, and «bf» for Behavior Fragment.
  • The implemented Context supports just one tag to represent situation, instead of AML's situationState and situationConstraint tags. An implemented situation tag is a string which specifies either the state name or the constraint text.
  • Iconic notation for Subset and Join is provided. Labels may be used for these stereotypes by switching the extended notation off.
  • Join selector is specified by means of the Branch property in StarUML.
  • Stereotypes «opredicate» and «oaction» were added to explicitly support modeling of the ontology concepts representing predicates and actions.
  • The player, roleProperty and roleType tags of the Create Role Action from AML are specified as one value of the DynamicArgument property in the textual format defined by AML.
  • Since multiplicity can be specified for a Classifier Role in StarUML, the multiplicity tag need not to be defined for Multi Lifeline.
  • As the payload of a Decoupled Message is specified by the message name, the payload tag is not supported by the implemented Decoupled Message. The same also applies for Communication Message.
  • Serviced Property is not explicitly supported. A property connected by Service Provisions or Service Usages is implicitly considered to be a Serviced Property. The same is true for Serviced Port.
  • Instead of a Mental Association, which is not explicitly supported, the AML profile for StarUML specifies the «mental ae» stereotype attached to a particular Association End that represents a Mental Property in association. The stereotype defines the same tags as Mental Property.

4. Known Issues

  • The AML diagram frame (called the Extended Frame in this implementation) can only be placed in sequence or collaboration diagrams. This is a restriction of StarUML.
  • Moving a stereotyped message or stimulus (Multi Message, Decoupled Message, Communication Stimulus, Join, etc.) may result in its incorrect placement in a Collaboration (Role) Diagram. The final position can be different than the position of dropping. This is caused by StarUML's positioning of the messages and stimuli drawn with extended notation. Workaround resides in switching the extended notation off, moving the message or stimulus at the desired position, and switching the extended notation on.
  • The AML profile should be included in a newly created model manually (by means of the Profile Manager dialog) in order to be used also after re-opening the model. Even if a new model has the AML profile included initially, the profile should be either (1) manually excluded and included again, or (2) included after next model opening. This problem is caused by StarUML which ignores inclusion of a "non-auto including" profile in newly created and saved models.
  • There may occur the problem of opening a StarUML model under the Windows system with the regional options set differently from the regional options used by the system during saving the file. This problem is identified by the error message "Could not convert variant of type (OleStr) into type (Double)" occurred during opening the model file. In order to fix this problem, a value of the Number format (or all Standards and formats settings) in the Regional and Language Options system dialog should be changed to the value used at the model saving time.

5. Further Improvements

The following improvements are foreseen:

  • On-line help providing information about AML elements and how to model them in StarUML.
  • Model checker to check the AML model for completeness and consistency as defined by the UML and AML specifications.
